<div class="entry-meta"><span class="posted-on"><span class="screen-reader-text">Posted on</span> <a href="http://www.wildml.com/2016/08/rnns-in-tensorflow-a-practical-guide-and-undocumented-features/" rel="bookmark"><time class="entry-date published" datetime="2016-08-21T11:10:51-07:00">August 21, 2016</time><time class="updated" datetime="2016-08-29T07:44:38-07:00">August 29, 2016</time></a></span><span class="byline"> by <span class="author vcard"><a class="url fn n" href="http://www.wildml.com/author/dennybritz/">Denny Britz</a></span></span></div><!-- .entry-meta --><h1 class="entry-title">RNNs in Tensorflow, a Practical Guide and Undocumented Features</h1>	</header><!-- .entry-header -->
<div class="entry-content">
<p>In a <a href="http://www.wildml.com/2015/09/recurrent-neural-networks-tutorial-part-1-introduction-to-rnns/">previous tutorial series</a> I went over some of the theory behind Recurrent Neural Networks (RNNs) and the implementation of a simple RNN from scratch. That&#8217;s a useful exercise, but in practice we use libraries like Tensorflow with high-level primitives for dealing with RNNs.</p>
<p>With that using an RNN should be as easy as calling a function, right? Unfortunately that&#8217;s not quite the case. In this post I want to go over some of the best practices for working with RNNs in Tensorflow, especially the functionality that isn&#8217;t well documented on the official site.</p>
<p>The post comes with a Github repository that contains Jupyter notebooks with minimal examples for:</p>
<ul>
<li><a href="https://github.com/dennybritz/tf-rnn/blob/master/sequence_example.ipynb">Using tf.SequenceExample</a></li>
<li><a href="https://github.com/dennybritz/tf-rnn/blob/master/batching_padding.ipynb">Batching and Padding</a></li>
<li><a href="https://github.com/dennybritz/tf-rnn/blob/master/dynamic_rnn.ipynb">Dynamic RNN</a></li>
<li><a href="https://github.com/dennybritz/tf-rnn/blob/master/bidirectional_rnn.ipynb">Bidirectional Dynamic RNN</a></li>
<li><a href="https://github.com/dennybritz/tf-rnn/blob/master/rnn_cell_wrappers.py.ipynb">RNN Cells and Cell Wrappers</a></li>
<li><a href="https://github.com/dennybritz/tf-rnn/blob/master/loss_masking.py.ipynb">Masking the Loss</a></li>
</ul>
<p><span id="more-820"></span></p>
<h3>Preprocessing Data: Use tf.SequenceExample</h3>
<p><strong><a href="https://github.com/dennybritz/tf-rnn/blob/master/sequence_example.ipynb">Check out the tf.SequenceExample Jupyter Notebook here!</a></strong></p>
<p>RNNs are used for sequential data that has inputs and/or outputs at multiple time steps. Tensorflow comes with a <a href="https://developers.google.com/protocol-buffers/">protocol buffer</a> definition to deal with such data: <a href="https://github.com/tensorflow/tensorflow/blob/master/tensorflow/core/example/example.proto">tf.SequenceExample</a>.</p>
<p>You can load data directly from your Python/Numpy arrays, but it&#8217;s probably in your best interest to use <code>tf.SequenceExample</code> instead. This data structure consists of a &#8220;context&#8221; for non-sequential features and &#8220;feature_lists&#8221; for sequential features. It&#8217;s somewhat verbose (it blew up my latest dataset by 10x), but it comes with a few benefits that are worth it:</p>
<ul>
<li><strong>Easy <a href="https://www.tensorflow.org/versions/r0.10/how_tos/distributed/index.html#distributed-tensorflow">distributed training</a>.</strong> Split up data into multiple TFRecord files, each containing many SequenceExamples, and use Tensorflow&#8217;s built-in support for distributed training.</li>
<li><strong>Reusability.</strong> Other people can re-use your model by bringing their own data into <code>tf.SequenceExample</code> format. No model code changes required.</li>
<li><strong>Use of Tensorflow data loading pipelines functions</strong> like <code>tf.parse_single_sequence_example</code>. Libraries like tf.learn also come with convenience function that expect you to feed data in protocol buffer format.</li>
<li><strong>Separation of data preprocessing and model code.</strong> Using <code>tf.SequenceExample</code> forces you to separate your data preprocessing and Tensorflow model code. This is good practice, as your model shouldn&#8217;t make any assumptions about the input data it gets.</li>
</ul>
<p>In practice, you write a little script that converts your data into <code>tf.SequenceExample</code> format and then writes one or more TFRecord files. These TFRecord files are parsed by Tensorflow to become the input to your model:</p>
<ol>
<li>Convert your data into <code>tf.SequenceExample</code> format </li>
<li>Write one or more TFRecord files with the serialized data</li>
<li>Use <a href="https://www.tensorflow.org/versions/r0.10/api_docs/python/io_ops.html#TFRecordReader"><code>tf.TFRecordReader</code></a> to read examples from the file</li>
<li>Parse each example using <a href="https://github.com/tensorflow/tensorflow/blob/r0.10/tensorflow/python/ops/parsing_ops.py"><code>tf.parse_single_sequence_example</code></a> (Not in the official docs yet)</li>
</ol>

<h3>Batching and Padding Data</h3>
<p><strong><a href="https://github.com/dennybritz/tf-rnn/blob/master/batching_padding.ipynb">Check out the Jupyer Notebook on Batching and Padding here!</a></strong></p>
<p>Tensorflow&#8217;s RNN functions expect a tensor of shape <code>[B, T, ...]</code> as input, where <code>B</code> is the batch size and <code>T</code> is the length in time of each input (e.g. the number of words in a sentence). The last dimensions depend on your data. Do you see a problem with this? Typically, not all sequences in a single batch are of the same length T, but in order to feed them into the RNN they must be. Usually that&#8217;s done by padding them: Appending <code>0</code>&#8216;s to examples to make them equal in length.</p>
<p>Now imagine that one of your sequences is of length 1000, but the average length is 20. If you pad all your examples to length 1000 that would be a huge waste of space (and computation time)! That&#8217;s where batch padding comes in. If you create batches of size 32, you only need to pad examples within the batch to the same length (the maximum length of examples in that batch). That way, a really long example will only affect a single batch, not all of your data.</p>
<p>That all sounds pretty messy to deal with. Luckily, Tensorflow has built-in support for batch padding. If you set <code>dynamic_pad=True</code> when calling  <a href="https://www.tensorflow.org/versions/r0.10/api_docs/python/io_ops.html#batch"><code>tf.train.batch</code></a> the returned batch will be automatically padded with 0s. Handy! A lower-level option is to use <a href="https://www.tensorflow.org/versions/r0.10/api_docs/python/io_ops.html#PaddingFIFOQueue">tf.PaddingFIFOQueue</a>.</p>
<h4>Side note: Be careful with 0&#8217;s in your vocabulary/classes</h4>
<p>If you have a classification problem and your input tensors contain class IDs (0, 1, 2, &#8230;) then you need to be careful with padding. Because you are padding tensos with 0&#8217;s you may not be able to tell the difference between 0-padding and &#8220;class 0&#8221;. Whether or not this can be a problem depends on what your model does, but if you want to be safe it&#8217;s a good idea to not use &#8220;class 0&#8221; and instead start with &#8220;class 1&#8221;. (An example of where this may become a problem is in masking the loss function. More on that later).</p>

<h3>rnn vs. dynamic_rnn</h3>
<p><strong><a href="https://github.com/dennybritz/tf-rnn/blob/master/dynamic_rnn.ipynb">Check out the dynamic_rnn Jupyter Notebook here!</a></strong></p>
<p>You may have noticed that Tensorflow contains two different functions for RNNs: <a href="https://www.tensorflow.org/versions/r0.10/api_docs/python/nn.html#rnn"><code>tf.nn.rnn</code></a> and <a href="https://www.tensorflow.org/versions/r0.10/api_docs/python/nn.html#dynamic_rnn"><code>tf.nn.dynamic_rnn</code></a>. Which one to use?</p>
<p>Internally, <code>tf.nn.rnn</code> creates an unrolled graph for a fixed RNN length. That means, if you call <code>tf.nn.rnn</code> with inputs having 200 time steps you are creating a static graph with 200 RNN steps. First, graph creation is slow. Second, you&#8217;re unable to pass in longer sequences (> 200) than you&#8217;ve originally specified.</p>
<p><code>tf.nn.dynamic_rnn</code> solves this. It uses a <code>tf.While</code> loop to dynamically construct the graph when it is executed. That means graph creation is faster and you can feed batches of variable size.  What about performance? You may think the static rnn is faster than its dynamic counterpart because it pre-builds the graph. In my experience that&#8217;s not the case.</p>
<p>In short, <strong>just use tf.nn.dynamic_rnn</strong>. There is no benefit to <code>tf.nn.rnn</code> and I wouldn&#8217;t be surprised if it was deprecated in the future.</p>
<h4>Passing sequence_length to your RNN</h4>
<p><strong><a href="https://github.com/dennybritz/tf-rnn/blob/master/bidirectional_rnn.ipynb">Check out the Jupyter Notebook on Bidirectional RNNs here!</a></strong></p>
<p>When using any of Tensorflow&#8217;s rnn functions with padded inputs it is important to pass the <code>sequence_length</code> parameter. In my opinion this parameter should be required, not optional. <code>sequence_length</code> serves two purposes: 1. Save computational time and 2. Ensure Correctness.</p>
<p>Let&#8217;s say you have a batch of two examples, one is of length 13, and the other of length 20. Each one is a vector of 128 numbers. The length 13 example is 0-padded to length 20. Then your RNN input tensor is of shape <code>[2, 20, 128]</code>. The dynamic_rnn function returns a tuple of (outputs, state), where <code>outputs</code> is a tensor of size <code>[2, 20, ...]</code> with the last dimension being the RNN output at each time step. <code>state</code> is the last state for each example, and it&#8217;s a tensor of size <code>[2, ...]</code> where the last dimension also depends on what kind of RNN cell you&#8217;re using.</p>
<p>So, here&#8217;s the problem: Once your reach time step 13, your first example in the batch is already &#8220;done&#8221; and you don&#8217;t want to perform any additional calculation on it. The second example isn&#8217;t and must go through the RNN until step 20. By passing <code>sequence_length=[13,20]</code> you tell Tensorflow to stop calculations for example 1 at step 13 and simply copy the state from time step 13 to the end. The output will be set to 0 for all time steps past 13. You&#8217;ve just saved some computational cost. But more importantly, if you didn&#8217;t pass <code>sequence_length</code> you would get incorrect results! Without passing <code>sequence_length</code>, Tensorflow will continue calculating the state until T=20 instead of simply copying the state from T=13. This means you would calculate the state using the padded elements, which is not what you want.</p>

<h3>Bidirectional RNNs</h3>
<p>When using a standard RNN to make predictions we are only taking the &#8220;past&#8221; into account. For certain tasks this makes sense (e.g. predicting the next word), but for some tasks it would be useful to take both the past and the future into account. Think of a tagging task, like part-of-speech tagging, where we want to assign a tag to each word in a sentence. Here we already know the full sequence of words, and for each word we want to take not only the words to the left (past) but also the words to the right (future) into account when making a prediction. Bidirectional RNNs do exactly that. A bidirectional RNN is a combination of two RNNs &#8211; one runs forward from &#8220;left to right&#8221; and one runs backward from &#8220;right to left&#8221;. These are commonly used for tagging tasks, or when we want to embed a sequence into a fixed-length vector (beyond the scope of this post).</p>
<p>Just like for standard RNNs, Tensorflow has static and dynamic versions of the bidirectional RNN. <h3>RNN Cells, Wrappers and Multi-Layer RNNs</h3>
<p><strong><a href="https://github.com/dennybritz/tf-rnn/blob/master/rnn_cell_wrappers.py.ipynb">Check out the Jupyter Notebook on RNN Cells here!</a></strong></p>
<p>All Tensorflow RNN functions take a <code>cell</code> argument. LSTMs and GRUs are the most commonly used cells, but there are many others, and not all of them are documented. Currently, the best way to get a sense of what cells are available is to look at at <a href="https://github.com/tensorflow/tensorflow/blob/master/tensorflow/python/ops/rnn_cell.py">rnn_cell.py</a> and <a href="https://github.com/tensorflow/tensorflow/blob/master/tensorflow/contrib/rnn/python/ops/rnn_cell.py">contrib/rnn_cell</a>.</p>
<p>As of the time of this writing, the basic RNN cells and wrappers are:</p>
<ul>
<li><code>BasicRNNCell</code> &#8211; A vanilla RNN cell.</li>
<li><code>GRUCell</code> &#8211; A <a href="http://arxiv.org/abs/1406.1078">Gated Recurrent Unit</a> cell. </li>
<li><code>BasicLSTMCell</code> &#8211; An LSTM cell based on <a href="http://arxiv.org/abs/1409.2329">Recurrent Neural Network Regularization</a>. No peephole connection or cell clipping.</li>
<li><code>LSTMCell</code> &#8211; A more complex LSTM cell that allows for optional peephole connections and cell clipping. </li>
<li><code>MultiRNNCell</code> &#8211; A wrapper to combine multiple cells into a multi-layer cell.</li>
<li><code>DropoutWrapper</code> &#8211; A wrapper to add dropout to input and/or output connections of a cell.</li>
</ul>
<p>and the contributed RNN cells and wrappers:</p>
<ul>
<li><code>CoupledInputForgetGateLSTMCell</code> &#8211; An extended <code>LSTMCell</code> that has coupled input and forget gates based on <a href="http://arxiv.org/abs/1503.04069">LSTM: A Search Space Odyssey</a>.</li>
<li>TimeFreqLSTMCell &#8211; Time-Frequency LSTM cell based on <a href="http://static.googleusercontent.com/media/research.google.com/en//pubs/archive/45401.pdf">Modeling Time-Frequency Patterns with LSTM vs. Convolutional Architectures for LVCSR Tasks</a></li>
<li><code>GridLSTMCell</code>Â &#8211; The cell from <a href="http://arxiv.org/abs/1507.01526">Grid Long Short-Term Memory</a>.</li>
<li><code>AttentionCellWrapper</code> &#8211; Adds attention to an existing RNN cell, based on <a href="https://arxiv.org/abs/1601.06733">Long Short-Term Memory-Networks for Machine Reading</a>.</li>
<li><code>LSTMBlockCell</code> &#8211; A faster version of the basic LSTM cell (Note: this one is in <a href="https://github.com/tensorflow/tensorflow/blob/master/tensorflow/contrib/rnn/python/ops/lstm_ops.py"><code>lstm_ops.py</code></a>)</li>
</ul>

<h3>Calculating sequence loss on padded examples</h3>
<p><strong><a href="https://github.com/dennybritz/tf-rnn/blob/master/loss_masking.py.ipynb">Check out the Jupyter Notebook on Calculating Loss here!</a></strong></p>
<p>For sequence prediction tasks we often want to make a prediction at each time step. For example, in Language Modeling we try to predict the next word for each word in a sentence. If all of your sequences are of the same length you can use Tensorflow&#8217;s <a href="https://github.com/tensorflow/tensorflow/blob/master/tensorflow/python/ops/seq2seq.py">sequence_loss and sequence_loss_by_example functions (undocumented)</a> to calculate the standard cross-entropy loss.</p>
<p>However, as of the time of this writing <code>sequence_loss</code> does not support variable-length sequences (like the ones you get from a dynamic_rnn). Naively calculating the loss at each time step doesn&#8217;t work because that would take into account the padded positions. The solution is to create a weight matrix that &#8220;masks out&#8221; the losses at padded positions.</p>
<p>Here you can see why 0-padding can be a problem when you also have a &#8220;0-class&#8221;. If that&#8217;s the case you cannotuse <code>tf.sign(tf.to_float(y))</code> to create a mask because that would mask out the &#8220;0-class&#8221; as well. You can still create a mask using the sequence length information, it just becomes more complicated.</p>
